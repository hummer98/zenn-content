## はじめに

**最初に断っておくと、cmux-team は実験的なプロジェクトです。** 「cmux で自律実行型のマルチエージェントを組んだらどうなるか？」を検証するために作ったもので、プロダクション品質のフレームワークではありません。設計判断も運用知見も、試行錯誤の途中経過です。

それを踏まえた上で、この記事では [cmux-team](https://github.com/hummer98/cmux-team) のアーキテクチャと、実際に動かして得た知見を共有します。

> この記事は [cmux エコシステム記事](https://zenn.dev/hummer/articles/cmux-ecosystem-claude-code) の cmux-team セクションを独立させたものです。

## クイックスタート

[Claude Code](https://docs.anthropic.com/en/docs/claude-code) と [cmux](https://cmux.dev) がインストール済みであることが前提です。

```bash
# 1. cmux 上で Claude Code を起動
# 2. Plugin をインストール
/plugin marketplace add hummer98/cmux-team
/plugin install cmux-team@hummer98-plugins

# 3. チーム起動
/cmux-team:start
```

これで Master と Manager が隣のペインに立ち上がります。あとは Master ペインに切り替えて、自然言語でタスクを伝えるだけです。

```
You:    TODO アプリを React で作って
Master: タスクを作成しました。Manager が検出して作業を開始します。
        → Manager がタスク検出 → Conductor を spawn
        → Conductor が Agent を起動、隣のペインで作業開始
        → 各 Agent の作業はリアルタイムで見える

You:    進捗は？
Master: Conductor-1: 実装中（Agent 2/3 完了）

You:    E2E テストも追加して
Master: 新しいタスクを作成しました。次のサイクルで着手します。
```

## なぜ作ったのか

### 既存ツールの課題感

マルチエージェント開発ツールはいくつか試しました。[Antigravity](https://github.com/antigravity-official/antigravity)、[Auto-Claude](https://github.com/mbenezit/auto-claude)、Claude Code の組み込み Agent ツールによるチーム構成など。どれも可能性は感じたのですが、共通して引っかかるポイントがありました。

- **サブエージェントのブラックボックス化** — Agent ツールで起動したサブエージェントが何をやっているか見えないのが不安でした。実際に途中で介入することは稀なんですが、「何をやろうとしているのか」がリアルタイムで見えるだけで安心感が全然違います。結果だけ返ってきて「ファイルを30個変更しました」と言われても、心穏やかではいられません
- **セッション間・プロジェクト間の通信手段がない** — Claude Code のセッション同士には IPC のような通信経路がなく、あるセッションが別のセッションに指示を出したり状態を問い合わせたりできない。シングルプロジェクト限定で協調させたくても、セッションを立ち上げるところまでは人間が手動でやるしかない
- **新規プロジェクトの立ち上げが手動** — いずれもプロジェクト/ディレクトリドリブンなので、実験用のリポジトリを用意したり、ワークスペースを準備したりする作業が人間の手に残りがちになる。

- **このターミナルが一体何をやってるのかわからなくなる** — ghostty + tmux でマルチプロジェクト運用していた時期もありましたが、ターミナルが増えると**今見ているペインが一体どのプロジェクトのどのタスクなのか**がわからなくなります。タブタイトルを手動で管理するのも限界でした。（一時期はissue番号を書いた付箋をモニタに貼ってました）

### インスピレーション

multi-agent-shogun に出会ったのが直接のきっかけです。

@[card](https://github.com/yohey-w/multi-agent-shogun)

![multi-agent-shogun の動作画面](https://github.com/yohey-w/multi-agent-shogun/raw/main/images/company-creed-all-panes.png =500x)

エージェント階層をターミナル上で可視化しながら自律協調させるというアプローチに「これだ」と感じました。自分でも cmux ベースで同じことをやってみたくなって、cmux-team を作り始めました。

## 設計思想：CLI ベースでベンダーロックインを避ける

自律型 AI コーディングエージェントは山ほどあります。[Devin](https://devin.ai/)、[Manus](https://manus.im/)、[OpenAI Codex](https://openai.com/index/codex/)――どれも「タスクを渡せば結果が返る」プロダクトとして完成度が高い。ただし共通して、**中で何が動いているか、どのモデルが使われるかはベンダー次第**です。来月もっといいモデルが出ても、ベンダーが対応するまで待つしかない。

cmux-team はプロダクトではなく**オーケストレーション層**です。下で動くエージェントが何であるかを関知しません。Manager が Conductor を起動するとき、やっていることは `cmux send` でターミナルにコマンド文字列を送っているだけ。

```bash
cmux send --surface "$SURFACE" "claude --dangerously-skip-permissions '...'\n"
```

ここを差し替えれば Gemini CLI でも OpenAI Codex でも動きます。Manager と Conductor が見ているのは「ターミナルの画面に `❯` が表示されたかどうか」だけなので、下で動くエージェントの種類は問いません。

この「何を起動するか」「どう結果を回収するか」をセマンティックに解決することで、気軽に新しいモデルやCLIツールを試せるというのも大きなメリットだと考えています。

```bash
# Claude の代わりに Gemini CLI を起動する例
cmux send --surface "$SURFACE" "gemini '...'\n"
```

もう一つの違いは**実行環境**です。Devin や Manus はコードがベンダーのサンドボックスで動きます。cmux-team は自分のターミナルで動くので、既存の開発環境――エディタ、デバッガ、shell 設定、社内ネットワーク、認証情報――がそのまま使えます。エージェントの入れ替えはテンプレート1行の変更で済みます。

## 何が変わったか

cmux-team（と using-cmux）を導入して、開発ワークフローにいくつかの変化がありました。

### Markdown 地獄からの解放

これは cmux-team というより [using-cmux](https://github.com/hummer98/using-cmux) の恩恵ですが、**他の surface の画面を AI のコンテキストとして直接読める**ようになったのが大きな変化です。

以前は、あるエージェントの状態を別のエージェントに伝えるために、スナップショット情報を Markdown ファイルに書き出しては読ませる、あるいは人間がプロンプトで説明する、という手順が必要でした。そして、この状態が変わるたびにファイルを更新して、読み込み側にも「ファイルを読み直して」と指示する。この Markdown 地獄から解放されました。

もちろんissue(+コメント)にして中継するのはとても良いアイデアですが、テンポラリな情報や、どこのプロジェクトに属するべきなのかよくわからないものまでissue化することになるので、**この狭間のような、形式化しにくいコンテキストをどう伝達するか**という問題がなかなか解決できませんでした。

```bash
# 別 surface の画面を直接読むだけ
cmux read-screen --surface surface:N --scrollback 200
```

実のところ、この「他のエージェントの動作状況」自体をコンテキストにするというニーズはあまり意識していませんでした。
そもそもこのエージェント間のコンテキストハンドオーバーを無意識的に避けるために、(過剰な)ドキュメントドリブンになりがちだったのですが、using-cmuxに脳を切り替えてからは、このAIのログ自体をコンテキストにする、というユースケースはかなり多い上に、AIエージェントチームの協調動作を指示する上で、非常に便利だということに気が付きました。
（claude-codeの全ログを対象にコンテキストとして食わせる事もありましたが、これはログを特定するだけでも時間とトークンを消費するのでかなり億劫でした）

### 作業キューによるプロジェクト多面打ち

Master にタスクを立て続けに指示できるようになったことで、**「次のタスクの指示」のために作業完了を待つ時間が減りました**（ゼロではないですが）。

```bash
# Master にタスクを連続投入
"ログイン機能を実装して" (taskが生成されてManagerが反応する)
"ログイン機能が実装されたらDiscordで報告してデプロイ" (Managerに報告タスクがキューされる)
"デプロイが終わったらsurface48に続行指示" ()
```

タスクファイルがセマンティックな作業キューとして機能するので、1プロジェクトへの指示をまとめて行えます。結果として、複数プロジェクトを同時進行する「多面打ち」のインターバルを伸ばせるようになりました。プロジェクト間のコンテキストスイッチ回数が減り、人間側の認知負荷が下がります。

### 非決定論的タスクの性能比較

複数のリポジトリ/ディレクトリの作成から実験結果の統合までを完全自律化できるので、**セマンティックな挙動の確認など、決定論的でないタスクの性能比較を気軽にできる**ようになりました。

例えば「プロンプト A と B でエージェントの出力品質がどう変わるか」を検証したいとき、それぞれの実験環境を自動構築して並列実行し、結果を統合するところまでを cmux-team に任せられます。

ただしトレードオフもあって、**トークン消費は爆発します**。マルチエージェントで並列実行すると、単体実行の数倍〜十数倍のトークンを消費するのは覚悟してください。

## 4層アーキテクチャ

cmux-team の核は、4つの層に分離されたエージェント階層です。

![cmux-team 4層アーキテクチャ](/images/cmux-team-4tier.jpeg =500x)

### 各層の責務

| 層 | 責務 | ライフサイクル |
|----|------|---------------|
| **Master** | ユーザーの指示を `.team/tasks/open/` にタスクファイルとして書き出す。進捗をユーザーに報告する。 | ユーザーセッション中は常駐 |
| **Manager** | タスクを検出して Conductor を起動。pull 型で進捗を監視し、完了したら結果を回収してタスクをクローズ。 | アイドル停止 + イベント駆動 |
| **Conductor** | 1タスクを git worktree 内で自律完遂。必要に応じて Agent を起動して並列作業。 | タスク完了で停止 |
| **Agent** | 実際のコード実装・テスト・リサーチを行う。 | 完了したら停止 |

### なぜ4層か

タスク管理と実作業を1つの層が兼任すると、2つの問題が起きます。

1. **スケジューリングが破綻する** — 並列タスクの進捗監視と実作業を同時にやろうとすると、どちらも中途半端になる
2. **ユーザーとの対話が途切れる** — 作業に集中すると、ユーザーへの報告が遅れる

4層に分離することで、各層が「自分の仕事だけ」に集中できます。Master は作業しない。Manager は監視だけ。Conductor は1タスクだけ。Agent は実作業だけ。

## イベント駆動の Manager

Manager は**アイドル停止 + イベント駆動**です。Master がタスクを作成すると `[TASK_CREATED]` 通知で起床し、Conductor を起動して監視。完了したら結果を回収してタスクをクローズし、未処理タスクがなければまたアイドルに戻ります。

## 実務層はアーキテクチャを知らなくていい

この設計で重要なのは、**実際に作業する Agent が cmux-team の全体アーキテクチャを一切知らなくてよい**ことです。

上位層（Manager / Conductor）が `cmux read-screen` で画面を読んで完了を検出する pull 型の通信なので、Agent 側は「作業して、終わったら止まる」だけ。上位への報告プロトコルも、タスク管理の仕組みも気にする必要がありません。

つまり、cmux-team の本質は特定のロール群ではなく、**既存の Agent やツールを自由に組み込んで、ワークフローをキック・監視する仕組み**です。普段使っている Claude Code のセッションや、Gemini CLI のスクリプトを、そのまま Agent として組み込めます。

## プリセットロール

cmux-team で固定なのは **Manager**（監視 + タスクディスパッチ）と **Conductor**（タスク実行 + Agent 管理）の2つだけです。

一応、その下で実際に動く実務Agent には、以下のプロンプトテンプレートがプリセットとして同梱されていますが、基本的には**あなたが使い慣れたAgent達に差し替えて**問題ありません。

| ロール | 用途 |
|--------|------|
| Researcher | 技術調査・情報収集 |
| Architect | 設計・アーキテクチャ決定 |
| Implementer | コード実装 |
| Tester | テスト作成・実行 |
| Reviewer | コードレビュー |
| DocKeeper | ドキュメント管理 |
| IssueManager | イシュー整理・優先度付け |

## 実際の運用で得た知見

### Manager のモデル選定

Manager は **Sonnet モデル**で動作します。Haiku だとプロトコルを無視して独自判断で動いたり、タスクファイルのフォーマットを守らなかったりしました。Opus は動作するもののコストパフォーマンスが悪い。Manager のような「プロトコルに忠実に従う」ロールには、指示追従性とコストのバランスが取れた Sonnet が最適でした。

### 権限制限の課題

Claude Code の `--dangerously-skip-permissions` は全ツールの権限チェックをスキップします。サブエージェントを自動起動するには必須ですが、権限の粒度が粗いのが悩みです。

理想的には「ファイル読み書きは許可、外部 API 呼び出しは都度確認」のような細かい制御がほしいところです。現状は `--dangerously-skip-permissions` を付けて、CLAUDE.md や Agent プロンプトで制約を書くしかありません。

### cmuxワークスペースのレイアウト対策

Agent を増やすとペインが狭くなりますが、`cmux-grid` コマンドで自動的にグリッド配置に整列できます。レイアウトが崩れたら `cmux-grid` や `cmux-grid 2x3` で修復してください。

### cmux 本体の未解決問題

cmux-team の運用で遭遇している cmux 本体側の issue を2つ紹介します。どちらもワークアラウンドで回避していますが、正直なところ力技です。

#### PTY 遅延初期化問題（[#1472](https://github.com/manaflow-ai/cmux/issues/1472)）

`cmux new-workspace` でプログラム的に作成したワークスペースのターミナル PTY は、**GUI 上で一度表示されるまで起動しない**という問題です。

バックグラウンドのワークスペースでは PTY（`ghostty_surface_t`）が生成されないため、`cmux read-screen` は `Surface not found` エラー、`cmux send-key` は `Surface not ready` エラーになります。`cmux send` は OK を返しますが、テキストはキューに留まって実際には配信されません。

ワークアラウンドとして、**AppleScript でメニュークリックしてワークスペースを GUI 描画させてから元に戻す**という荒業をやっています。macOS のアクセシビリティ許可が必要で、画面もチラつきます。cmux 本体側でヘッドレスな PTY 生成がサポートされれば不要になるはずですが、現状はこの方法しかありません。

この問題があるため、cmux-team では `new-workspace` よりも同一ワークスペース内の `new-split` を優先するレイアウト戦略を採っています。

#### drag-surface-to-split の cross-workspace 問題（[#1901](https://github.com/manaflow-ai/cmux/issues/1901)）

`cmux drag-surface-to-split` は V1 ソケットハンドラを経由しており、surface の解決に**現在 GUI でフォーカスされているワークスペース**を使います。そのため、呼び出し元と異なるワークスペースの surface を操作しようとすると `Surface not found` エラーになります。

`new-split` などの他のコマンドは V2 ハンドラで `--workspace` パラメータを受け付けるため cross-workspace で動作しますが、`drag-surface-to-split` だけが取り残されている状態です。

ワークアラウンドとしては、**`drag-surface-to-split` を使わずに `new-split` + `cmux send` で代替する**か、操作前に `select-workspace` で対象ワークスペースにフォーカスを移してから実行しています。

## おわりに

AI開発界隈にいらっしゃるみなさんにはすでにご承知のとおり、現在のAI基盤モデルおよび関連ツール群はあまりに進化が早すぎでして、従来のフルパッケージされているような統合環境は非常に便利ではあるものの、陳腐化の速度も激しいために特定の環境やワークフローにロックインするのは非常にギャンブルな状況だと思います。

また、自律実行型のエージェントに関しては、OpenClawをはじめその強力さが喧伝される以上に、ユーザー側の不安感が強く、導入に二の足を踏むケースが多いのが実情だと思います。

「アウトプットさえ正しければ良い」という割り切りができるのは一部の職種だけで、プロプライエタリなコードを扱う業務環境に導入するのはハードルが高いのが実情だと思います。

`cmux-team`は、「AIエージェントのバランスの良い可視化」「ツール/ベンダーロックインからの離脱」という志向を持っているのですが、現状のAIツール群が***本当に向き合うべき問題***は、機能面の話ではなく

**「使う側の不安感をどうやったら取り除けるのか？」**

という、多分にエンパシック(Empathic)な観点でのUXデザイン問題だと考えています。

cmux-teamがその"不安解消"の一助となれば幸いです。
